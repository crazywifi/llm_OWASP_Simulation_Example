📐 ML08: Model Skewing

Exploiting Training-Production Data Drift

🎯 Attacker's Goal

Manipulate the production environment or input distribution to cause model behavior to deviate from training expectations. By exploiting the difference between what the model was trained on versus what it encounters in production, attackers can cause systematic failures without modifying the model itself.

📋 Step 1: Training Data Distribution

The model was trained on data with a specific distribution. It performs well when production data matches this distribution.

Expected Production

Assumption: ML models assume production data will resemble training data. When this assumption breaks, so does model performance.

⚔️ Step 2: Inducing Distribution Shift

Attackers manipulate the production environment to create data drift.

🌐 Environment Manipulation

Change lighting, camera angles, or sensor conditions to shift input distribution.

📊 Input Flooding

Send many requests from unusual distribution to bias online-learning models.

⏰ Temporal Drift

Exploit natural changes over time that weren't captured in training.

🎭 Feature Covariate Shift

Subtle changes to input features that push model out of comfort zone.

💥 Step 3: Skewed Distribution Attack

🌍 Step 4: Real-World Attack Scenarios

🚗 Autonomous Vehicles

Scenario:

Model trained on sunny California roads. Attacker operates in foggy conditions or nighttime, causing perception failures.

💳 Fraud Detection

Scenario:

Fraudsters adapt behavior to appear more like legitimate users, exploiting distribution assumptions.

📧 Spam Filters

Scenario:

Spammers evolve language and formatting to shift away from known spam patterns.

🏥 Medical Imaging

Scenario:

Different scanner manufacturers or patient demographics cause model degradation.

🔍 Step 5: Why Skewing is Hard to Detect

🎭 Silent Degradation

Model doesn't crash - it just becomes less accurate. No obvious errors to trigger alerts.

📊 Gradual Shift

Changes may be slow and subtle, looking like natural variation.

🏷️ No Ground Truth

In production, you often don't have labels to verify model correctness.

🌊 Concept Drift Confusion

Hard to distinguish malicious skew from natural concept drift.

🛡️ Step 6: Developer Recommendations

✅ Defense Strategies

  • Distribution Monitoring: Continuously track input feature distributions
  • Drift Detection: Implement KL divergence, PSI, or other drift metrics
  • Model Uncertainty: Track confidence scores - drops indicate potential skew
  • Diverse Training Data: Train on varied conditions to improve robustness
  • A/B Testing: Compare model versions to detect degradation
  • Feedback Loops: Collect production labels to validate predictions
  • Retraining Pipelines: Regularly retrain on recent data
  • Fallback Systems: Have rule-based backups for high-uncertainty cases

📖 Key Takeaway

ML models are only as good as their training data assumptions. Monitor production inputs continuously, detect drift early, and build systems that gracefully handle distribution shifts rather than silently failing.
